# 项目背景

## 原始问题

这个项目来自一个很具体的工作流问题：日常工作中大量事务都被“紧急排查”牵引，尤其是线上问题、群里被艾特的问题、需要查数据库或反复验证的问题。每个问题本身都需要等待、查询、验证和来回沟通，单个排查过程经常要十几二十分钟才能推进一步。

为了不在等待中空转，用户会同时开启多个问题排查：每个问题建一个 worktree、切一个分支、查一段、等一段，再切到另一个问题。这个方式短期能填满等待时间，但长期会带来三个副作用：

- 并行问题太多，当前到底在排查什么变得混乱。
- 每条线索散在聊天、截图、数据库结果和不同 worktree 里，恢复上下文成本高。
- 工作节奏被紧急问题牵着走，越来越难主动规划和沟通。

## Will 的建议

和 Will 讨论后的核心建议是：不要只沿着“紧急”这一条线工作，要主动保留一条“重要但不一定有今天期限”的线。

典型例子：

- 梳理 agent 仓库，提升对代码结构的熟悉度。
- 优化排查工作流，减少每次定位问题的等待和切换成本。
- 探索可替代或辅助工具，在现有工具慢的时候有备用路线。

这些事情不是每天都会被人催，但做好后会直接降低未来紧急排查的耗时，所以它们是“重要”的。项目需要帮助用户把这类事务从紧急噪音里捞出来，并在每天早上固定时间先处理一段，除非真有必须立刻响应的事项。

## 产品目标

这个工具不是普通待办清单，而是一个本机工作事务面板，目标是帮助用户同时处理两条线：

- 紧急排查线：有明确期限或外部压力的问题，按状态、截止日期和序号推进。
- 重要建设线：没有立即期限，但能长期改善工作效率和主动性的事务。

面板每天打开后应该回答三个问题：

1. 现在有哪些任务还没完成，今天自然结转过来了？
2. 哪些任务有时限压力，需要按序号推进或控制并行数量？
3. 哪些重要但无期限的任务需要被保护，不要一直被排查事项挤掉？

## 为什么需要后端和图片

排查类任务经常不只有文字，还会包含报错截图、数据库结果、聊天上下文、接口返回、页面状态等材料。只靠纯文本会丢上下文，下一次接着查时仍然要回聊天或终端里翻。

因此项目从一开始就需要：

- 后端保存任务和图片，而不是只存在浏览器内存里。
- 支持粘贴截图，减少记录成本。
- 图片能在卡片上直接看到缩略图，必要时复制给同事或贴回聊天。
- 任务备注可以直接露出一部分，让卡片本身承担上下文恢复功能。

## 模型演进

最初设想是传统四象限：紧急度和重要性两个维度，可能用 1-10 分数表达。但实际使用后发现，紧急度分数容易变成主观维护成本：

- “明天前要做”和“现在马上要做”都属于有时限，只是日期不同。
- 用户真正需要的是截止时间自动影响排序，而不是每天手动调紧急度分数。
- AI 对 1-10 打分不稳定，但判断“重要吗”和“哪天截止”更稳定。

所以当前模型改为两个更清晰的问题：

- `important`：这件事重要吗？
- `due_date`：哪天截止？没有期限就为空。

四象限由这两个字段派生：

- 重要 + 无期限：长期建设、上午保护时间优先做。
- 重要 + 有期限：主要排查和交付任务，按过期、状态、截止日期和手动序号推进。
- 不重要 + 无期限：少做或不做。
- 不重要 + 有期限：穿插快速处理。

旧 `urgency / importance` 打分列只保留数据库兼容，新代码不再读写。

## 关键设计原则

- 任务不属于某一天。每天的面板是查询结果，靠 `created_date` 和 `completed_date` 实现自动结转与历史回放。
- 有期限任务不需要定时任务上浮。打开面板时按今天日期重新排序，过了一夜“明天”自然变“今天”。
- `review` 和 `verify` 都不是完成。PR 待审、等人看、已合并待真实环境验证时，任务应留在面板上，不进入归档。
- AI 只生成草稿，不直接入库。用户需要在预填弹窗里确认、修改、贴图后再保存。
- 工具首先服务单人本机使用，优先轻量可靠，而不是提前引入账号、多端同步或复杂权限。

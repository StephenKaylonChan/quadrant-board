## 问题清单

### P0 - 结构性问题

无 P0。当前项目规模和单人本机定位下，没有发现需要立即大拆架构的问题。

### P1 - 实现质量问题

1. [D9] `frontend/package.json:6-10`、`backend/requirements.txt:1-6` [置信度:高]  
   描述: 没有前端测试框架、后端测试框架或交互回归脚本。  
   依据: 当前脚本只有 `dev/build/preview`，后端依赖没有 pytest/httpx 测试依赖。拖拽、图片复制、历史回放、状态归档都只能手测。

2. [D2] `frontend/src/components/TaskEditor.tsx:54-251` [置信度:高]  
   描述: 编辑弹窗职责过多，已经是 486 行组件。  
   依据: 同一组件内处理表单状态、图片上传、临时 URL 生命周期、删除、未保存确认、快捷键、灯箱入口和 API 调用。后续继续加字段或交互时修改风险会升高。

3. [D2] `frontend/src/App.tsx:71-185` [置信度:中]  
   描述: 顶层 App 状态容器继续承担业务格式化和快捷键逻辑，正在向“万能入口”增长。  
   依据: `buildDailySync`、删除确认、同步弹窗复制、主题监听、加载和拖拽乐观更新都在 App 中。

4. [D8] `backend/app/routers/tasks.py:29-40` [置信度:中]  
   描述: 每日面板接口全量返回当天所有任务，没有分页、状态筛选或轻量字段模式。  
   依据: 查询只按日期过滤并返回完整 `TaskOut` 与 `images`；本机小数据可接受，但大量历史任务和图片时前端渲染会变重。

5. [D8] `frontend/src/components/QuadrantBoard.tsx:113-116` [置信度:中]  
   描述: 每次渲染都对所有任务按四个象限重复 filter，并对 active 列表排序。  
   依据: 当前数据量小问题不大；但图片缩略图、拖拽状态、同步弹窗等状态变化都可能触发重复计算。可用 `useMemo` 或按象限预分组降低重算。

6. [D6] `backend/app/routers/ai.py:104` [置信度:中]  
   描述: AI 响应结构假设较强，非标准响应可能抛出未转译异常。  
   依据: 直接读取 `resp.json()["choices"][0]["message"]["content"]`；如果兼容接口返回错误 JSON、空 choices 或字段变化，用户会看到 500 而不是中文 502。

7. [D6/D11] `frontend/src/clipboard.ts:7-16`、`frontend/src/components/TaskCard.tsx:63-70`、`frontend/src/components/Lightbox.tsx:24-31` [置信度:高]  
   描述: 图片复制依赖 `createImageBitmap`、`ClipboardItem` 和剪贴板写权限，跨浏览器失败路径不统一。  
   依据: Firefox/Safari 对图片写入剪贴板支持不一致；卡片缩略图复制失败静默忽略，灯箱会显示失败。建议统一能力探测和 fallback 文案。

8. [D11/可访问性] `frontend/src/components/TaskCard.tsx:79-100` [置信度:高]  
   描述: 卡片是可点击、可拖拽的 `article`，但没有键盘打开入口或语义角色。  
   依据: 鼠标点击可编辑，键盘用户无法 tab 到卡片打开；`draggable` 也不是可靠的键盘交互替代。

9. [D11/跨浏览器] `frontend/src/components/TaskCard.tsx:81-100` [置信度:中]  
   描述: 依赖 HTML5 `draggable`，移动端和触屏浏览器支持不稳定。  
   依据: 目前只有 mouse drag/drop 事件，没有 pointer/touch fallback；移动端只能退化为编辑字段改象限。

### P2 - 卫生与可维护性问题

10. [D5] `frontend/src/App.tsx:16-30`、`frontend/src/components/TaskCard.tsx:20-34`、`frontend/src/components/TaskEditor.tsx:23-29` [置信度:高]  
    描述: 任务状态的 label、icon、同步分组、编辑选项分散在三个文件。  
    依据: 新增 `verify` 时需要多处同步；后续改状态名、排序、图标容易漏改。建议抽 `statusMeta.ts`。

11. [D5] `frontend/src/components/TaskEditor.tsx:109-120`、`frontend/src/components/TaskEditor.tsx:213-251`、`frontend/src/App.tsx:159-185`、`frontend/src/components/Lightbox.tsx:12-20` [置信度:中]  
    描述: 全局事件监听模式重复，且 `TaskEditor` 两个 effect 没有依赖数组，每次渲染都会重新绑定再清理。  
    依据: 不是内存泄漏，但认知负担高；建议提取 `useDocumentEvent` 或给 effect 明确依赖。

12. [D7] `frontend/src/clipboard.ts:11` [置信度:中]  
    描述: 使用非空断言 `canvas.getContext('2d')!`。  
    依据: 正常浏览器通常成立；但如果 canvas context 获取失败，会抛出非业务化错误。建议显式判断并返回中文错误。

13. [D7] `frontend/src/components/QuadrantBoard.tsx:130`、`frontend/src/components/TaskCard.tsx:96` [置信度:中]  
    描述: `relatedTarget as Node | null` 类型断言重复。  
    依据: 当前可工作；但这是浏览器事件边界，可抽一个小 helper，减少断言散落。

14. [D11] `frontend/src/styles.css:1-115`、`frontend/src/styles.css:338-710` [置信度:高]  
    描述: CSS 文件承担主题 token、布局、卡片、弹窗、灯箱、动效等所有职责，超过 1000 行。  
    依据: 已经成为视觉改动热点。建议不引入 CSS Modules 的前提下，至少用注释分区和变量表继续治理；中期可拆 `styles/tokens.css`、`styles/board.css`、`styles/modal.css`。

15. [D11/视觉] `frontend/src/styles.css:11`、`frontend/src/styles.css:296` [置信度:中]  
    描述: 浅色主题 accent 小字对浅底对比度约 3.94:1，低于普通小字号 AA 4.5:1。  
    依据: `--accent #c75233` 在浅纸色背景上用于 `AI` 12px 标签等小字。建议小字使用 `--accent-deep` 或加深一档。

16. [D11/视觉] `frontend/src/styles.css:33-34` [置信度:中]  
    描述: 浅色主题“不重要 · 无期限”的标题色对底色约 4.28:1，略低于普通文本 AA 4.5:1。  
    依据: `#756d5e` on `#edebe2`，标题字号 16px 可能勉强可读，但小屏/低亮度下偏灰。

17. [D11/跨浏览器] `frontend/src/styles.css:663-667` [置信度:中]  
    描述: 备注两行截断只使用 `-webkit-line-clamp`，缺少标准 `line-clamp` 或无 clamp fallback 策略。  
    依据: Chromium/Safari 支持较好，Firefox 新版支持情况仍需验证；不同浏览器可能显示行数不一致。

18. [D11/可视性] `frontend/src/styles.css:672-682` [置信度:中]  
    描述: 卡片图片横向滚动在 Chrome/Safari 上没有自定义 scrollbar，滚动可发现性偏弱。  
    依据: 只设置了 Firefox 的 `scrollbar-width/color`；用户可能不知道还能横向滑，尤其触控板以外环境。

19. [D6] `frontend/src/api.ts:5-13` [置信度:中]  
    描述: 前端请求层只校验 HTTP 状态，不校验成功响应结构。  
    依据: `res.json()` 直接断言为泛型 `T`；后端字段变更或代理返回 HTML 时会在组件层才暴露异常。

20. [D8] `backend/app/routers/tasks.py:108-115` [置信度:中]  
    描述: 上传图片逐张完整读入内存，多图同时上传时内存峰值可达 `图片数 × 10MB`。  
    依据: 单张 10MB 限制合理，但接口允许多文件，且每张 `await file.read()` 后再写盘。

### P3 - 观察项

1. `frontend/src/App.tsx:54-68`：今日同步生成逻辑现在简单可靠；如果后续加入“待确认/需求/PR 链接”字段，应避免继续堆在 App，改为 util。
2. `frontend/src/components/QuadrantBoard.tsx:50-63`：排序逻辑清晰，但一旦状态继续增加，应与状态 meta 合并维护。

5. `frontend/src/components/AiQuickAdd.tsx:40-42`：Enter 直接拆任务，对单行输入合理；如果未来改多行输入需重新设计快捷键。
6. `backend/app/routers/tasks.py:81-89`：删除任务时磁盘文件删除失败不会上报；本机工具可以接受，后续可加孤儿文件清理。
7. `frontend/index.html` favicon 使用 data URL，简单有效；若以后做 PWA 或多尺寸图标再换成静态资源。

## 不做（投入产出比低）

1. 后端立即拆 Service/Repository：不做。  
   理由: 当前后端 9 个源文件、router 逻辑清晰，架构文档也明确单人本机工具可先集中在 router。拆层收益低，反而增加跳转。

2. 引入成熟拖拽库：暂不做。  
   理由: 当前需求是本机桌面使用，HTML drag/drop 已满足基础链路。真正要支持移动端/触摸时再评估 `dnd-kit` 或 pointer 方案。

3. 立即建设完整设计系统：不做。  
   理由: 无 UI 库、手写 CSS 是既定选择；现在更适合先整理 token、状态 meta 和视觉回归 checklist。

## 跨边界观察

- 前后端 `status` 已同步为五档，但前端状态展示信息散落，后端只是 Literal。下一步若继续增加状态，应先抽共享语义文档或前端 status meta，避免再做多点同步。
